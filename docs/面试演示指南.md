### 5. 架构讲解（5 分钟）

#### 5.1 六边形架构（Python AI 服务）

```
┌──────────────────────┐
│  Primary Ports       │  ← REST API / gRPC
│  (Driving Adapters)  │
├──────────────────────┤
│  Core Domain         │  ← GraphRAG, Multi-Agent, LLM
├──────────────────────┤
│  Secondary Ports     │  ← ES, Milvus, MinIO
│  (Driven Adapters)   │
└──────────────────────┘
```

**话术：**
> "Python 服务使用六边形架构，核心域不依赖外部实现，
> 所有外部服务通过适配器接入，支持 Mock/真实切换。"

#### 5.2 COLA 架构（Java 业务服务）

> "Java 服务使用阿里巴巴 COLA 4.x 架构，
> 分为 Adapter、Application、Domain、Infrastructure 四层。"

#### 5.3 技术亮点

- **LLM 适配器设计模式**：模型无关的抽象层，支持 Ollama/OpenAI/Mock 自由切换
- **GraphRAG**：知识图谱增强检索，发现传统 RAG 无法发现的关联风险
- **多模态支持**：MinIO 存储 + 多模态 LLM 分析
- **LangGraph 工作流**：状态机编排 RiskAgent → SearchAgent
- **多级 Fallback**：LLM 失败 → 规则引擎，保证系统可用性
- **模型微调**：LoRA 微调 + GGUF 导出

## 常见问题准备

### Q: 为什么用 GraphRAG 而不是传统 RAG？

A: 传统 RAG 只能检索独立文档片段，无法理解实体间的关系。
在安保场景中，一个位置的事件往往会影响关联区域。
GraphRAG 通过知识图谱发现隐含的关联风险。

### Q: 数据从哪来？

A: 公开数据集（中国应急管理预案）+ LLM 生成扩展数据 + 手动构建知识图谱。

### Q: 如果 LLM 服务挂了怎么办？

A: 多级 Fallback：LLM 不可用时自动降级到规则引擎，保证系统可用性。

### Q: 模型怎么微调的？

A: 使用 LoRA 低秩适配，从种子数据生成 instruction-tuning 格式数据，
用 unsloth/LLaMA-Factory 训练，导出 GGUF 格式供 Ollama 使用。
