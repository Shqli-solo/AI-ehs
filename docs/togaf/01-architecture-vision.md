## 2. 问题陈述

### 2.1 业务问题
当前 EHS 系统存在以下核心问题：
1. **响应延迟高**: 事故响应时间 45min，无法满足生产安全要求
2. **决策准确率低**: 复杂场景问答准确率 82%，误报/漏报率高
3. **人工依赖重**: 需要大量人工复核，工作效率低
4. **系统孤岛**: 各子系统独立运行，缺乏统一决策中台

### 2.2 技术债务
- 缺乏统一的架构设计文档
- 服务间通信协议未标准化
- 缺少完整的监控和可观测性
- 部署流程手工化，缺乏 CI/CD

---

## 3. 项目目标

### 3.1 业务目标 (SMART 原则)
| 指标 | 当前值 | 目标值 | 提升 |
|------|--------|--------|------|
| 事故响应时间 | 45min | 8min | 82% 提升 |
| 问答准确率 | 82% | 95% | 13 个百分点 |
| 人工复核工作量 | 100% | 30% | 减少 70% |
| 年度规避损失 | - | 1500 万+ | 新增 |

### 3.2 技术目标
1. **架构现代化**: 实现 Monorepo + 六边形架构 + 微服务
2. **AI 能力升级**: 集成真实 LLM + 微调模型 (指令/风险/Embedding)
3. **工程卓越**: 测试覆盖>80%, CI/CD 自动化，K8s 容器化部署
4. **可观测性**: Prometheus+Grafana 监控，LangFuse 链路追踪

## 4. 架构原则

### 4.1 核心原则
| 编号 | 原则 | 理由 | 示例 |
|------|------|------|------|
| P1 | 分离关注点 | 降低复杂度，提高可维护性 | 六边形架构、COLA 四层 |
| P2 | 设计降级 | 生产环境外部依赖可能不可用 | 检索器降级处理 |
| P3 | 可观测性优先 | 生产问题快速定位 | 指标/日志/追踪三位一体 |
| P4 | 自动化优先 | 减少人工干预 | CI/CD、IaC |
| P5 | 安全内置 | 安全不是附加功能 | JWT 认证、TLS、输入验证 |

### 4.2 技术选型原则
- **前端**: Next.js 14 + TypeScript + Shadcn/UI (SSR + 类型安全 + 现代设计)
- **AI 服务**: Python 3.11 + FastAPI + LangGraph (AI 生态丰富)
- **业务服务**: Java 17 + Spring Boot + COLA (企业级稳定性)
- **通信协议**: REST (前端通信) + gRPC (服务间高效通信)
- **部署**: Docker + K8s Helm (容器编排标准)

## 6. 转型挑战

### 6.1 技术挑战
| 挑战 | 影响 | 缓解措施 |
|------|------|----------|
| 微调数据准备 | 数据质量影响模型效果 | LLM 生成 + 交叉验证 + 人工校验 |
| 服务间通信 | 循环依赖风险 | 明确服务边界，Java→Python 单向调用 |
| gRPC TLS 配置 | 双向认证复杂度高 | 预留证书配置文档，提前验证 |
| K8s Helm 部署 | 配置复杂度高 | 参考官方 Chart，分阶段验证 |
